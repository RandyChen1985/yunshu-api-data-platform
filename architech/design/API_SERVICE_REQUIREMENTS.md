## 2. 项目概述

### 2.1 项目背景

当前云枢・智维・AI平台的数据访问主要通过内部接口实现，缺乏标准化的API服务。为了支持业务方的数据消费需求，需要将现有的数仓库表进行服务化改造，提供标准化的API接口。

### 2.2 项目目标

- 构建一个面向多数据源（ClickHouse、HBase、MySQL 等）的通用数据服务平台
- 将现有及后续新增的数仓 / 业务库表服务化，提供标准化 API 接口
- 支持多种查询方式（分页、筛选、聚合、统计、时间序列）
- 提供统一的认证授权与审计机制，支持角色 + 数据资源两层权限控制
- 保证 API 的性能、稳定性与水平扩展能力
- 提供完整的 API 文档、SDK 与示例，方便云枢 Web 及其他系统复用

### 2.3 项目范围

- 通用数据访问 API：对外暴露统一的 RESTful 接口，屏蔽 ClickHouse / HBase / MySQL 等底层差异
- 统一认证授权体系：基于 Token/JWT 的认证、角色与数据资源两层访问控制
- API 网关和路由管理：统一入口、限流、熔断与灰度发布
- API 监控和日志系统：统一请求链路追踪与审计日志
- API 文档和开发者门户：统一的接口说明、在线调试与示例代码
- 为上层应用（如云枢 Web、AI Agent、第三方系统）提供稳定的 Data API 能力

## 3. 需求分析

### 3.1 功能性需求

#### 3.1.1 数据查询API

- **REQ-API-001**：提供基础数据查询接口，支持按条件筛选
- **REQ-API-002**：提供分页查询功能，支持大数据量分页
- **REQ-API-003**：提供数据聚合查询功能（统计、求和、平均值等）
- **REQ-API-004**：提供时间序列数据查询功能
- **REQ-API-005**：MVP 阶段不支持多表关联查询（JOIN / 跨源关联）；后续版本按数据源能力逐步支持

#### 3.1.2 接口形态（两套并行）

- **REQ-API-SHAPE-001**：同时提供两类 API：
  - **资源化接口**：面向常用业务数据/表的固定路径（例如动环实时指标、动环事件等）
  - **通用查询接口**：面向上层系统提供统一的查询入口（例如 `LogicalQuery`），由服务端完成路由与执行
- **REQ-API-SHAPE-002**：两类接口使用不同的 path 前缀，避免语义混淆与迭代耦合

#### 3.1.3 认证授权

- **REQ-AUTH-001**：提供基于 API Key 的认证机制
- **REQ-AUTH-002**：支持API访问权限控制
- **REQ-AUTH-003**：支持访问频率限制
- **REQ-AUTH-004**：提供API访问日志记录

#### 3.1.4 API管理

- **REQ-MGMT-001**：提供API版本管理功能
- **REQ-MGMT-002**：支持API访问统计
- **REQ-MGMT-003**：提供API性能监控
- **REQ-MGMT-004**：支持API访问告警

## 6. 接口规范

### 6.1 通用规范

- 使用 RESTful API 设计风格
- 数据格式统一采用 JSON
- 所有接口响应 MUST 使用统一 Envelope 结构：`{ code, message, data, timestamp, trace_id }`
- `code` 用于表示业务状态（200 为成功），`message` 为人类可读提示，`data` 为业务负载
- `timestamp` 记录服务端时间，`trace_id` 用于链路追踪（贯穿 API 服务与 AI 服务）
- 统一错误码规范，所有异常均映射为上述 Envelope 结构

#### 6.1.1 分页响应规范（统一）

- 对于分页类接口，`data` 字段固定为对象结构：
  - `items`: 数组
  - `total`: 总记录数
  - `page`: 当前页（从 1 开始）
  - `size`: 每页数量
  - `pages`: 总页数

示例：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "size": 20,
    "pages": 0
  },
  "timestamp": "2025-12-26T11:00:00+08:00",
  "trace_id": "..."
}
```

### 6.2 认证规范

- 统一使用 **API Key** 鉴权
- 客户端在请求头中携带 `X-API-Key: <api_key>`
- `api_service` 需要提供 API Key 的生命周期管理能力（创建/禁用/轮换/查询）
- `X-Scopes`（可选）：预留给上游透传数据范围（MVP 阶段不做强校验，仅记录与透传）

### 6.3 错误处理

- 统一错误响应格式
- 标准错误码定义
- 详细的错误信息描述

## 7. 权限管理

### 7.1 权限级别

- **角色层（Role）**：按照用户身份划分，如管理员（Admin）、领导层（Leader）、运维工程师（Operator）、观察者（Viewer）等。
  - 角色层决定可调用的 API 范围以及是否允许跨区域 / 跨机房访问，优先级高于资源层。
- **资源层（Scope）**：与具体数据资源绑定，如区域（region）、机房（room）、机架（rack）等。
  - 例如：`regions: ["华东","华北"], rooms: ["ROOM_A","ROOM_B"]`。

### 7.2 访问控制

- 支持上游在请求中透传 `role` / `scopes` 等用户上下文信息（例如通过 `X-Scopes`）
- MVP 阶段不对 `scopes` 做强校验与过滤，仅用于日志审计与后续扩展预留
- 支持数据脱敏策略（例如按角色对敏感字段进行掩码或禁止访问）

> 说明：MVP 阶段 `scopes` 先以 `X-Scopes` 透传为主，不做强校验；后续版本再落地资源层权限过滤。
