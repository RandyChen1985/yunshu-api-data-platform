## Context

目前系统数据接口存在敏感信息泄露风险。我们需要实现一套类似流控系统的三级配置脱敏机制（全局 -> 角色 -> 用户）。

## Goals / Non-Goals

**Goals:**
1.  **三级覆盖策略**：优先级为 `用户 > 角色 > 全局`。
2.  **动态配置与示例说明**：配置界面提供详细的匹配规则说明和脱敏效果预览。
3.  **高性能**：通过规则缓存和字段匹配结果缓存，将脱敏开销控制在毫秒级。
4.  **默认初始化**：预置通用脱敏规则（手机号、邮箱、密码、身份证）。

## Decisions

### 1. 三级配置逻辑
- **全局级**: 系统总开关，定义默认行为。
- **角色级**: `GLOBAL` (跟随全局), `ENABLE` (强制脱敏), `DISABLE` (允许明文)。
- **用户级**: `ROLE` (跟随角色), `ENABLE` (强制脱敏), `DISABLE` (允许明文)。

### 2. 数据库变更
- `sys_masking_rules`: 核心规则表。
- `sys_roles.masking_strategy`: 存储角色级策略。
- `sys_users.masking_strategy`: 存储用户级策略。
- `sys_config`: 存储全局开关 `ENABLE_DATA_MASKING`。

### 3. 脱敏匹配优化
使用 `O(1)` 的哈希表记录已匹配过的字段名，避免在同一请求的大数据量循环中重复进行正则匹配。

### 4. 前端交互
在配置页面加入示例（Example）展示。例如：
- 字段名：`*phone*`
- 算法：`PARTIAL_DISPLAY_3_4`
- 预览：`13800138000` -> `138****8000`

## Risks / Trade-offs

- **[Risk] 性能风险**: 大数据量递归。
  - **Mitigation**: 引入字段名缓存，确保同一请求内每类字段仅匹配一次。
- **[Risk] 规则冲突**: 多个通配符规则匹配同一个字段。
  - **Mitigation**: 采用“首次匹配原则”，按规则的优先级/顺序执行。
