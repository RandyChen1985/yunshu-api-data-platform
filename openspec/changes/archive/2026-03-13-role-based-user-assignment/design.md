## 上下文 (Context)
系统通过 `sys_user_role_relation` 表维护用户与角色的多对多关联。
目前管理员只能在“用户编辑”中操作，效率较低。当角色发生变动或需要按职能批量授权时，需要一种“以角色为锚点”的操作模式。

## 目标与非目标 (Goals / Non-Goals)

**目标：**
- 在“角色管理”界面提供直观的成员管理入口。
- 支持批量选择用户并将其绑定到指定角色。
- 保持与现有 `sys_user_role_relation` 表结构的完全兼容。
- 确保操作后相关用户的权限缓存能及时失效并重建。

**非目标：**
- 不涉及用户基本信息（用户名、状态等）的修改。
- 不涉及角色权限（菜单/按钮/资源）的修改（已有专门接口）。

## 技术决策 (Decisions)

### 1. API 接口设计
在 `management.py` 中新增两个端点：
- **GET `/roles/{role_id}/users`**: 获取当前属于该角色的用户 ID 列表。
- **PUT `/roles/{role_id}/users`**: 批量更新该角色的成员。采用“全量覆盖”策略，即传入的 ID 列表即为该角色最终的成员列表。
  - **原因**：全量覆盖策略相比增量添加/删除更易于保持前后端状态一致，且实现逻辑简单稳健。

### 2. 前端交互方案
- **入口**: 在 `Roles.vue` 表格的操作列增加一个“成员管理”图标按钮。
- **组件**: 弹出对话框，内部使用 `Transfer` (穿梭框) 组件。
  - 左侧：系统中所有活跃用户。
  - 右侧：当前已关联到该角色的用户。
- **数据加载**: 打开弹窗时，并行调用用户列表接口和角色成员接口。

### 3. 缓存一致性
- **方案**: 批量更新成功后，调用 `PermissionService.invalidate_role_cache(role_id)`。
  - **原因**：确保受影响的用户在下一次请求时能立即应用新的权限集合。

## 风险与权衡 (Risks / Trade-offs)

- **[风险] 数据量过大** → 如果用户量达到数万，穿梭框一次性加载会导致卡顿。
- **[对策]** 当前系统用户量预期在千级以内，暂不引入分页选择。未来可升级为带搜索功能的远程选择器。
