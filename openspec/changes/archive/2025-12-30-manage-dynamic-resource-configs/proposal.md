# 提案：动态资源配置管理

**变更 ID**: `manage-dynamic-resource-configs`
**状态**: `草稿 (Draft)`

## 目标 (Goal)
将 `ClickHouseAdapter` 中硬编码的 `RESOURCE_CONFIGS` 迁移到基于数据库的动态元数据管理系统中。这将实现数据资源的标准化“元管理”，允许在不修改代码的情况下添加或更新资源映射。

## 问题 (Problem)
当前，新增一个数据表或修改 API 接口的字段映射，都需要修改 `app/services/data_adapter/clickhouse.py` 中的代码。这种方式不够灵活，即使是简单的配置更新也需要完整的发版部署流程。

## 解决方案 (Solution)
1.  **数据库存储**: 将资源配置存储在 MySQL 表 (`sys_resource_meta`) 中，并设计通用字段以支持多种数据源（不仅限于 ClickHouse）。
2.  **管理 API**: 提供 REST API 接口用于配置的增删改查。
3.  **前端管理界面**: 开发 Vue 前端页面，提供可视化的资源配置管理功能，支持表单验证和测试连接。
4.  **动态加载框架**: 重构数据适配层，使其支持通过 `data_source` 类型动态路由到不同的适配器（ClickHouse, MySQL, HBase 等），并从数据库加载相应配置。
5.  **Schema 校验**: 确保存储的元数据符合预定义的 Schema 规范。

## 影响 (Impact)
- **后端**: 新增 `MetaService`，重构适配器工厂模式以支持动态类型，数据库新增表。
- **前端**: 新增 "资源管理" 模块，包含列表页和编辑详情页。
- **运维**: 具备无需发版即可接入新数据源（如 MySQL 业务表）的能力。

## 风险 (Risks)
- **性能**: 每次请求都查库获取配置会有性能损耗。 **缓解措施**: 使用本地 LRU 缓存或 Redis，并设置较短的过期时间 (TTL)。
- **Schema 漂移**: 数据库中的配置可能与实际 ClickHouse 表结构不一致。 **缓解措施**: 提供校验 API 来检查一致性。
