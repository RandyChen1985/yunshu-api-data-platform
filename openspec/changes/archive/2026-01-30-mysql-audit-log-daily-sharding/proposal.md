# Proposal: MySQL 审计日志按日物理分表 (方案 A)

## 现状 (Problem)
审计日志单表过大（500w+），导致 DML 操作缓慢，索引维护成本高。现有的扩展表设计增加了查询复杂度。需要一种物理上彻底隔离、清理效率最高（Drop Table）的方案。

## 目标 (Goals)
- 每天生成一张独立的日志表，表名格式：`api_access_logs_yyyyMMdd`。
- 将审计扩展字段直接存入分表，消除 Join 需求。
- 提供动态建表机制，支持业务无感切换。
- 支持小范围内的跨日期查询（通过动态 UNION ALL）。

## 方案大纲 (High-level Plan)
1. **模板表定义**：创建 `api_access_logs_template` 作为克隆基准。
2. **中间件重构**：修改写入逻辑，实现“表名动态路由”和“自动按需建表”。
3. **查询层工具类**：封装一个 SQL 构建器，根据时间范围自动检测并拼接存在的物理分表。
4. **清理自动化**：修改清理脚本，扫描并删除过期的物理表。

## 风险评估
- **跨天查询性能**：大范围跨天查询会产生大量 UNION ALL，需在界面上限制查询跨度（如建议单次查询不超过 7 天）。
- **DDL 频率**：每天只需执行一次建表操作，通过内存 Cache 可消除重复的建表检查开销。
