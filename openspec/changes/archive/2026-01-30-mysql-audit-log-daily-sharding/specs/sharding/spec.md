# Spec: 审计日志分表存储

## ADDED Requirements

### R1: 动态分表写入
系统必须根据请求发生的日期，将审计日志写入对应的分表中。
#### Scenario: 跨天日志自动切换
- **Given**: 系统日期从 2026-01-30 跨越到 2026-01-31。
- **When**: 2026-01-31 00:00:01 收到请求。
- **Then**: 系统应自动创建 `api_access_logs_20260131`（如果不存在）并将日志存入该表，而非 `api_access_logs_20260130`。

### R2: 审计字段整合
分表结构必须包含原始日志的所有字段以及扩展表的审计字段。
#### Scenario: 记录 SQL 审计信息
- **Given**: 一个触发了 SQL 执行的 API 请求。
- **When**: 中间件捕获到 `state.source_sql`。
- **Then**: 该 SQL 语句应直接存储在分表的 `source_sql` 字段中，无需关联 `api_access_logs_ext`。

### R3: 性能优化 (DDL 缓存)
系统应避免在每次请求写入时都执行数据库元数据检查。
#### Scenario: 内存缓存命中
- **Given**: 当天已成功写入过一条日志，表已创建。
- **When**: 第二条日志写入。
- **Then**: 系统应通过内存 Cache 直接确定表已存在，直接执行 `INSERT`。
